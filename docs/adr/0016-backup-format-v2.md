# ADR 0016 — Backup format v2: per-install PBKDF2 salt

**Status**: Proposed
**Date**: 2026-05-07
**Author**: navidrast (drafted following v1.0-rc audit finding [H])
**Builds on**: ADR 0012 (backup-and-export)
**Code touched (when accepted)**: `internal/backup/cipher.go`,
`internal/backup/store.go`, `internal/backup/cipher_test.go`,
plus pre-v2-fixture restore tests.

## Context

The v1.0-rc audit flagged the backup encryption's PBKDF2 salt as a
hardening opportunity:

> Backup encryption uses a globally-static PBKDF2 salt
> (`"opendray-v1-backup"`, `internal/backup/cipher.go:46,96`). Two
> installs with the same `OPENDRAY_BACKUP_KEY` produce identical key
> material. Per-install random salt (stored alongside backup metadata,
> NOT secret) hardens against rainbow-table-style attacks and would
> also let multi-tenant deployments share infra without sharing keys.
> AES-256-GCM + 200k PBKDF2 iters + per-frame nonce + frame-index AAD
> are otherwise well-designed.

The everything-else-good caveat is important: the rest of the design
is sound. This ADR is narrowly scoped to **the salt**.

### What's actually at risk today

Threat model A — **shared passphrase across installs**:

If two operators (or two installs run by the same operator) share an
`OPENDRAY_BACKUP_KEY`, they end up with the **identical AES-256 key**.
A backup taken on install X can be opened by a Cipher built on
install Y. This is normally a non-event (single-admin self-host) but
becomes confusing in:

- Lab → prod parity testing where the same passphrase is reused
- Multi-tenant deployments where each tenant should be cryptograph-
  ically isolated even when the operator's secret manager hands out
  a shared seed
- Disaster recovery where an old key file mysteriously decrypts a
  newer install's backup the operator wasn't expecting

Per-install salt makes "same passphrase, different install" produce
**different keys**, even when the rest of the world is identical.

Threat model B — **offline cracking with stolen ciphertexts**:

If an attacker exfiltrates encrypted backup files plus the install's
static salt (which is a literal in source — public the moment the
repo flips public), they can run a passphrase dictionary against
PBKDF2-HMAC-SHA256 at 200,000 iterations. With `OPENDRAY_BACKUP_KEY`
generated by `openssl rand -base64 32` the practical attack cost is
≫ 2¹⁹⁸ regardless of salt — entropy of the passphrase dominates.
Per-install salt buys little here.

The dominant motivation is **threat model A** — install isolation.
Operators expect "different install = different key" by default.

### Why this hasn't shipped yet

The change is **format-breaking** unless done carefully. Wrong
implementation = silently unreadable backups. The audit's
"per-install random salt" recommendation is one sentence; the
implementation has at least four binding decisions (below) and
hard restore-compat obligations.

This ADR locks those decisions before any code touches `cipher.go`.

## Decisions

### D1 — Where does the salt live?

| Option | Pros | Cons |
|---|---|---|
| **In-header (chosen)** | Self-contained: any backup file can be decrypted with just the file + the passphrase. Cold-restore from a stolen disk works without DB access. | +16 bytes per backup file. Salt is visible alongside the ciphertext (acceptable — salt is **not** secret). |
| In DB | Backup file is unchanged; salt lives in `backup_targets` or `backup_runs` row. | Cold-restore needs DB up. Disaster scenarios where DB is gone become harder. **Disqualifies the option.** |

**Decision: In-header.** Backup must be self-contained. The audit
explicitly says "stored alongside backup metadata, NOT secret" —
in-header satisfies both.

### D2 — How is the format version bumped?

| Option | Pros | Cons |
|---|---|---|
| **Bump version byte 1→2 (chosen)** | Tiny diff. Magic stays `ODBK`. Existing readers reject v2 with a clear "version 2" error today, so a new tool decoding old binary is the only failure path. | Requires the `header[4]` byte check at `cipher.go:240` to switch on version. |
| New magic (`ODBK2` or similar) | Old code refuses to even start parsing. | Bigger blast radius, two parallel parser paths. |

**Decision: Bump version byte.** v1 → v2.

### D3 — Header v2 layout

```
v1 (current):
  bytes [0..3]   magic "ODBK"
  byte  [4]      version = 1
  bytes [5..7]   reserved (3 bytes of zero)
  total          8 bytes

v2 (proposed):
  bytes [0..3]   magic "ODBK"
  byte  [4]      version = 2
  bytes [5..7]   reserved (3 bytes of zero)
  bytes [8..23]  salt (16 random bytes from crypto/rand)
  total          24 bytes
```

16 bytes is sufficient for a salt — same length as the GCM nonce,
~2¹²⁸ space, no rainbow table fits. Rationale matches RFC 8018 §4
("at least 64 bits, but choose more").

### D4 — Dual-decode policy

| File version | Cipher built with v2-capable code | Cipher built with v1-only code (older binary) |
|---|---|---|
| v1 file | Decode using static salt `"opendray-v1-backup"`. Log INFO once per process: "decoded v1 backup; consider creating a v2 backup to migrate." | Decode using static salt (current behaviour). |
| v2 file | Read 16-byte salt from header[8..23], derive key from passphrase + salt, decode. | Reject with `ErrCipherFormat: version 2`. |

**v1 backups are decryptable forever.** No EOL date for v1 — they
just become "old format that still works." This is non-negotiable;
operators have years of v1 backups by the time this ships.

### D5 — When does a Cipher emit v1 vs v2?

A Cipher created via `NewCipher(passphrase)` writes v2 by default.

`NewCipher` gains an option to write v1 for compat-test fixtures:

```go
NewCipher(passphrase string, opts ...Option) (Cipher, error)

// e.g. backup.WithLegacyV1Format() to force v1 writes for fixture
// generation. Production code never calls it.
```

### D6 — Salt provenance per file

The salt is generated **per Seal call**, not per Cipher. Two backups
from the same Cipher have different salts and therefore different
derived keys (even though the passphrase is the same). This means
the AEAD `c.aead` and the Fingerprint() must be **lazy / per-stream**
in v2 — they can't be pre-baked at NewCipher time.

`Fingerprint()` semantics change:
- v1: `SHA-256(staticKey)[:8]` — one fingerprint per Cipher.
- v2: `SHA-256(salt || passphrase)[:8]` — per-file fingerprint.
  `Fingerprint()` returns empty until the first Seal/Open completes.
  Callers that store fingerprint per backup row already do this
  correctly (see `backup/store.go`); a Cipher-wide fingerprint
  becomes meaningless under v2 and is removed.

**This is the biggest behavioural change.** The current contract
"Fingerprint of a Cipher == fingerprint of every backup it produces"
breaks. Audit the call sites:

```bash
grep -rn "\.Fingerprint()" internal/backup/
```

Every site must read the fingerprint from the backup file's record,
not from the Cipher.

### D7 — Field-level encryption (`EncryptField`/`DecryptField`)

Field encryption is used for short payloads inside JSON
(`backup_targets.config` SMB passwords, summarizer API keys reusing
this cipher per ADR 0013). It is NOT framed and has no header.

**Decision: leave field-level encryption on the static salt
indefinitely.** Field envelopes carry `v1:` prefix; bumping that is
a separate ADR. Threat model A doesn't apply (field ciphertexts
aren't standalone backup files; they're rows in the same DB).

## Rollout

Three small PRs, in order:

### PR-1 — read v2, still write v1 (read-only enablement)

- `cipher.go` learns to dispatch on `header[4]` byte.
- v2 read path implemented; v2 write path stubbed behind
  `WithFormatV2()` test-only option.
- Tests:
  - Round-trip v1 backup (existing tests, regression guard).
  - Round-trip v2 backup via `WithFormatV2()` test option.
  - Open v2 file written by an external test fixture.
  - Refuse v3 (`header[4] = 3`) cleanly with `ErrCipherFormat`.

After PR-1, a Cipher running this code can decode any v1 OR v2
backup. Existing operators see no change — Seal still writes v1.

### PR-2 — switch the default to v2

- Default `NewCipher` writes v2.
- `backup/store.go`'s `Fingerprint()` reads from the persisted
  `backup_runs.fingerprint` column (already exists per ADR 0012)
  rather than the Cipher.
- Old per-Cipher Fingerprint() method removed; explicit error
  message points at the row column for callers that hadn't migrated.
- Tests:
  - Every existing backup-flow test now produces v2 by default.
  - Cross-install isolation test: two `NewCipher` instances built
    from the same passphrase produce different salts → different
    keys → install X cannot decode install Y's backup.
  - `WithLegacyV1Format()` option still produces a working v1 file.

### PR-3 — fixture & restore-from-old guarantees

- `internal/backup/testdata/v1-backup.bin` — pre-generated v1 fixture
  checked into the repo. Generated once, never regenerated.
- `TestOpen_V1Fixture_StaysReadable_Forever` — opens the fixture,
  verifies plaintext round-trip. Lives at `cipher_test.go`. **This
  test is the spec.** If it ever fails, v1-compat is broken.
- Documentation in `internal/backup/cipher.go` package-level comment
  noting the v1 fixture is the canonical compat anchor.

## Restore-compat test obligations

Beyond the v1 fixture above:

1. **Real-DB restore test** — Linivek runs once, manually:
   - Take a v1 backup of a populated DB on the current binary
     (against pre-PR-1 main).
   - Apply PR-1 + PR-2.
   - Restore the v1 backup against an empty DB.
   - Verify row counts match and a sampled set of rows is identical.

2. **Field-level cipher untouched** — automated test that
   `EncryptField/DecryptField` round-trips against the v1 envelope
   format unchanged. (Field cipher is OUT of scope per D7.)

3. **Cross-install test** — automated. Two Ciphers, same passphrase,
   different salt → file from one cannot be opened by the other,
   AND the open call returns `ErrCipherWrongKey` (not `ErrCipherFormat`
   or `ErrCipherCorrupted`).

## Consequences

**Good:**

- Each install has cryptographically independent backups even when
  passphrases are accidentally shared.
- File format becomes self-describing — operators with just a backup
  file and a passphrase can decrypt without ANY OpenDray state.
- The 16 extra bytes per file are negligible (a single backup is
  typically 10-100MB+).

**Trade-offs:**

- `Fingerprint()` semantics change (D6). Every call site needs an
  audit. Compile errors surface this — but only after PR-2 ships.
- Two parser paths in `cipher.go` indefinitely. The v1 path costs
  ~50 LOC + the v1 fixture test, forever.
- The static salt in `kdfSalt` becomes a frozen v1 protocol
  constant. Don't ever rename or "modernise" it — it's load-
  bearing for v1-format backups.

**Out of scope (future ADRs):**

- Key rotation (an operator wants to change `OPENDRAY_BACKUP_KEY`
  without losing access to old backups).
- Algorithm change (post-quantum cipher swap).
- Per-target-kind cipher policy (e.g. local backups encrypted but
  S3 backups doubly-encrypted with KMS).

## Decision

**Proposed.** Awaiting Linivek review. Implementation requires a Go
toolchain to validate against pre-existing v1 backups; this ADR
captures the design so the implementation PRs (PR-1 / PR-2 / PR-3
above) can be small and reviewable.

When accepted, update Status to `Accepted, <date>` and reference
this ADR from the PRs that ship the format change.
